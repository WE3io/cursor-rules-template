# Change Acceptance Criteria

Quality gate for canonical updates to `ai-blindspots/` (rules, articles, quick reference). A candidate change is ready for inclusion when it passes this checklist.

## Checklist

### Clarity

- [ ] Intent is stated clearly (what problem does this solve?)
- [ ] Target audience is obvious (who is this for?)
- [ ] No ambiguous or conflicting wording
- [ ] Jargon is explained or linked

### Actionability

- [ ] Reader can act on the guidance (concrete steps or examples)
- [ ] Constraints are explicit (when does this apply?)
- [ ] Outcomes are observable (how do we know it worked?)

### Testability

- [ ] Can be verified (examples, scenarios, or acceptance criteria)
- [ ] Edge cases considered (when might this not apply?)
- [ ] No untestable claims

### Portability

- [ ] Tool-agnostic where possible (intent, not mechanism)
- [ ] Can be translated to Cursor, Claude, Codex, Gemini, Antigravity
- [ ] No tool-specific assumptions unless scoped

### Evidence Quality

- [ ] Sources cited where applicable
- [ ] Confidence level indicated (high/medium/low)
- [ ] Rationale provided (why does this work?)
- [ ] Risk of ignoring is stated

## Usage

1. Apply when evaluating intake (BL-003) or proposed edits
2. Use with triage (BL-005) for go/no-go
3. Record decision in BL-004 format
